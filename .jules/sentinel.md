## 2025-02-09 - Path Traversal in Secondary Endpoints
**Vulnerability:** The `/api/jobs` endpoint accepted an `input_filename` (and `bg_image` if relative) directly from a JSON payload and joined it with `UPLOAD_FOLDER` without any sanitization, allowing arbitrary file read/deletion (e.g. `../../../etc/passwd` or traversing to delete critical files) via path traversal.
**Learning:** Developers correctly secured the primary file upload endpoint `/api/upload` using `secure_filename`, but failed to realize that secondary endpoints relying on client-supplied filenames for subsequent actions (like starting a job) MUST independently sanitize those inputs. Relying purely on the assumption that clients will only send "safe" filenames returned from the upload endpoint is a common flaw.
**Prevention:** Always validate and sanitize file paths constructed from user-supplied data, regardless of whether the endpoint is a primary upload or a secondary processing step. Consistently use `werkzeug.utils.secure_filename` on ALL filenames received in JSON payloads or query parameters before using them in `os.path.join`.
