## 2024-05-17 - Secondary Endpoint Path Traversal via JSON Payload
**Vulnerability:** The secondary `/api/jobs` endpoint was vulnerable to path traversal and arbitrary file reads/writes because it relied entirely on the primary `/api/upload` endpoint for file sanitization. The `/api/jobs` endpoint accepted a JSON payload with a `filename` property and directly concatenated it with the upload folder, allowing an attacker to spoof the filename and target files outside the intended directory.
**Learning:** Even if a primary upload endpoint securely sanitizes filenames, secondary endpoints that process or retrieve those files must independently validate and sanitize the references (filenames or paths) provided via client requests, as the client can easily spoof references to unintended files.
**Prevention:** Always use `werkzeug.utils.secure_filename` (or an equivalent strict validation mechanism) on ANY file name or path component received from a client payload, regardless of whether the file was supposedly uploaded through a secure process previously. Do not assume client-provided identifiers are trustworthy.
## 2025-02-26 - Secondary JSON Endpoint Path Traversal
**Vulnerability:** The `/jobs` endpoint in `web_app/api.py` accepted unsanitized `bg_image` absolute paths and `original_filename` parameters via JSON requests. An attacker could bypass the initial `/upload` sanitization by spoofing these JSON parameters to perform path traversal for arbitrary file deletion during job cleanup, or constructing malicious output paths.
**Learning:** Initial sanitization at the primary upload boundary is insufficient if secondary execution endpoints blindly trust client-provided file paths or filenames in JSON payloads.
**Prevention:** Always independently sanitize file names and paths using `werkzeug.utils.secure_filename` at every endpoint that handles them, especially when reconstructing paths or deleting files, regardless of previous upload steps. Never trust absolute paths or relative traversals provided by the client.
## 2026-05-21 - Flask Debug Mode Exposed on Public Interface
**Vulnerability:** The Flask application in `web_app/app.py` and `web_app/run.py` was bound to the `0.0.0.0` interface with `debug=True`. This exposed the Werkzeug interactive debugger, which allows arbitrary Python code execution, to anyone on the network.
**Learning:** Never use `debug=True` when exposing a Flask development server to non-localhost interfaces (e.g., `host='0.0.0.0'`). This instantly turns a helpful developer tool into a critical remote code execution vulnerability.
**Prevention:** Always set `debug=False` when binding to `0.0.0.0`, or bind only to `127.0.0.1` during debugging. In production, use a proper WSGI server (like Gunicorn) where debug mode is not applicable.
